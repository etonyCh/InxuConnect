1. Présentation du Projet
1.1 Contexte et vision
InzuConnect (« inzu » signifie « maison » en Kirundi) est une plateforme de mise en relation entre
hôtes et voyageurs au Burundi. Elle combine la confiance communautaire, les paiements mobile
money, l'accessibilité basse consommation de données et l'intelligence artificielle pour créer le premier
marketplace de logement conçu pour et par les Burundais.
Le projet part du constat que le marché burundais est radicalement différent des marchés où Airbnb a
émergé :
•
•
•
•
•

Seulement 12,5 % des adultes ont un compte bancaire formel (contre 34 % en Afrique subsaharienne).
Le taux de pénétration mobile est de 66 % (8,6 M abonnés pour 13 M habitants en 2023).
La pénétration internet est de 11,3 % début 2024 — connectivité faible et intermittente.
Les wallets EcoCash et Lumicash dominent les transactions numériques (16,8 % de pénétration
MFS en 2022).
La confiance interpersonnelle prime sur les systèmes institutionnels.

1.2 Problème résolu
Aujourd'hui, la location de logements au Burundi se fait principalement via des groupes Facebook,
WhatsApp et le bouche-à-oreille. Ces canaux souffrent de :
•
•
•
•

Absence de vérification des hôtes et des voyageurs → arnaques fréquentes.
Paiements non sécurisés, litiges sans recours.
Aucune visibilité sur la disponibilité réelle, les équipements (groupe électrogène, Starlink, etc.).
Friction extrême : il faut parfois plusieurs jours de négociation pour une réservation.

1.3 Proposition de valeur
Cible

Valeur créée

Voyageur (Guest)

Trouver, payer et réserver un logement en moins de 60 secondes, sans carte
bancaire, même avec un réseau faible.

Hôte (Host)

Lister sa chambre en 3 minutes, recevoir l'argent en 2 heures, sans formation
technique.

Agent communautaire

Gagner une commission en aidant les voisins à s'inscrire et gérer leurs
annonces.

ONG / Entreprises

Portail B2B pour loger les équipes en mission, avec facturation consolidée.

1.4 Objectifs
1. Atteindre 500 hôtes actifs et 2 000 réservations en 6 mois post-lancement à Bujumbura.
2. Déploiement dans 3 villes (Bujumbura, Gitega, Ngozi) en 12 mois.
3. Taux de complétion de réservation > 80 % (vs ~30 % pour les canaux informels actuels).

3. Périmètre Fonctionnel
3.1 MVP — Phase 1 (Mois 1-3)
Objectif : valider l'adoption avec le minimum de fonctionnalités permettant une réservation
complète et un paiement réel.

3.1.1 Module Authentification & Profil
•
•
•
•

Inscription par numéro de téléphone (OTP SMS via Africa's Talking ou Twilio).
Vérification d'identité : photo CNI / passeport + selfie (liveness check via API tiers : Smile
Identity ou Onfido Africa).
Profil bilingue Kirundi/Français : nom, photo, ville, langues parlées.
Badge de confiance progressif : Vérifié → Fiable → SuperHost.

3.1.2 Module Listings (Annonces)
•
•
•
•
•
•
•

Création d'annonce en 3 étapes : photos → infos → prix.
Upload de 3 à 10 photos compressées côté client (max 800 KB/photo).
Amélioration automatique des photos par IA (luminosité, contraste, recadrage).
Génération automatique de description en Kirundi et Français via LLM à partir des champs
remplis par l'hôte.
Champs spécifiques Burundi : groupe électrogène (oui/non), citerne d'eau, gardien, distance
taxi-moto, Starlink Wi-Fi, cuisine disponible.
Tarification en BIF (Franc Burundais) uniquement.
Calendrier de disponibilité visuel.

3.1.3 Module Recherche & Découverte
•
•
•
•
•

Recherche textuelle : « inzu Bujumbura 20 000 FBu ».
Filtres locaux : ville / quartier, prix min-max BIF, équipements (générateur, WiFi, eau), type
(chambre / maison / appartement).
Tri par distance, prix, note.
Carte interactive (Mapbox, fond de carte allégé).
USSD fallback : *303# → recherche basique, résultats par SMS.

3.1.4 Module Réservation & Paiement
•
•
•
•
•
•
•

Sélection de dates + nombre de voyageurs.
Récapitulatif : prix nuit × durée + frais service + surcharge générateur (si applicable).
Paiement via EcoCash, Lumicash, virement bancaire.
Option « Acompte maintenant, solde à l'arrivée » (50/50).
Reçu automatique par SMS et notification push.
Réservation instantanée ou « Demande hôte » selon préférence de l'hôte.
Escrow : le paiement est débité mais retenu jusqu'à confirmation d'arrivée (check-in scan QR).

4. Exigences Non Fonctionnelles
4.1 Performance
•
•
•
•

Temps de chargement de la page d'accueil < 3 secondes sur réseau 3G (10 Mbps simulé).
Taille de l'APK Android < 8 MB (sans assets téléchargeables).
Cache offline : dernières 10 recherches, annonces favorites, conversations actives.
Images servies en WebP, dimensionnées côté serveur (CDN).

4.2 Disponibilité & Scalabilité
•
•
•

SLA cible : 99,5 % uptime (hors maintenance planifiée).
Architecture horizontalement scalable — déploiement conteneurisé.
Déploiement sur infrastructure africaine en priorité (ex. AWS af-south-1 Cape Town ou serveur
local si disponible).

4.3 Sécurité
•
•
•
•
•

Chiffrement en transit : TLS 1.3 obligatoire.
Chiffrement au repos : AES-256 pour les données sensibles (CNI, selfies, numéros de
téléphone).
Conformité RGPD-like : droit à l'effacement, portabilité des données, consentement explicite.
Authentification JWT avec refresh token rotatif (durée de vie access token : 15 min).
Rate limiting API : 100 req/min par IP, 1 000 req/min par compte business.

4.4 Accessibilité & Localisation
•
•
•
•

Langue par défaut : Kirundi. Seconde langue : Français. Troisième : Anglais (diaspora).
Swahili pour les zones frontalières (Bujumbura port, Uvira).
Police de caractères lisible sur écrans basse résolution (minimum 14px corps de texte).
Mode nuit activable (économie batterie sur appareils bas de gamme).

Principes directeurs : technologies matures, faible coût opérationnel, recrutement possible

Composant

i18next + react-i18next — support Kirundi, Français, Anglais, Swahili.

6. Architecture Système
6.1 Vue d'ensemble
L'architecture adopte un pattern modulaire monolithique pour le MVP (plus simple à déployer et à
maintenir avec une petite équipe), avec des frontières de domaine claires permettant une migration
vers des microservices si la croissance le justifie.

Couche Client : React Native App (Android/iOS) ←→ Next.js Web App
↕ HTTPS / WebSocket
Couche API : Fastify API Server (TypeScript) ← BullMQ Workers
↕
Couche Données: PostgreSQL + PostGIS | Redis (cache/queue) | R2 (media)
↕
Services tiers: Africa's Talking | InTouch | Smile Identity | Claude API | Mapbox
